# RAGENT MySQL Schema Report

本文件说明 `RAGENT` 数据库中已经导入的 ADS 与 DWS V4 表结构。

当前版本只保留：

- ADS 层：4 张表
- DWS 层：只选用 V4 版本，6 张表

数据来源：

- `/home/chenjy/r-data-agent/data-worker-design-master/sql/ads`
- `/home/chenjy/r-data-agent/data-worker-design-master/sql/dws/v4`

转换说明：

- 原始 SQL 是 Doris 方言。
- 当前已转换为 MySQL/InnoDB 可执行表结构。
- Doris 的 `ENGINE=OLAP`、`AGGREGATE KEY`、`DISTRIBUTED BY`、`AUTO PARTITION`、`PROPERTIES` 等能力已移除。
- Doris 的 `BITMAP`、`HLL` 字段在 MySQL 中转换为 `LONGBLOB`。
- 这些 `LONGBLOB` 字段用于保留结构含义，但在 MySQL 中不具备 Doris Bitmap 原生计算能力。
- `ads_query_examples.sql` 是查询示例文件，不包含建表语句，因此未导入。

## 整体分层理解

### DWS V4 层

DWS 是明细汇总层，负责保存更接近事实过程的聚合数据。

本项目选用 DWS V4，是因为 V4 已经把旧版多个 DWS 表进行了整合：

- `dws_v4_user_daily`：用户视角，每日/小时登录、注册、创角、付费。
- `dws_v4_device_daily`：设备视角，每日/小时激活、新设备、买量/自然量拆分。
- `dws_v4_ad_cost_daily`：广告花费视角，每日广告消耗、曝光、点击、转化。
- `dws_v4_cohort_user`：用户队列视角，D0 到 D360 留存、付费留存、LTV。
- `dws_v4_cohort_device`：设备队列视角，D0 到 D360 设备留存、设备 LTV、刷量风险指标。
- `dws_v4_user_segment`：用户分群视角，保存分群标签与用户集合。

DWS V4 更适合回答细粒度问题，例如：

- 某一天某个小时的 DAU 是多少？
- 某个广告素材带来的新增用户、付费用户、首日收入是多少？
- 某个队列的 D7、D30、D90 留存如何？
- 用户视角和设备视角的新增是否一致？
- 哪些设备存在多账号风险？

### ADS 层

ADS 是应用服务层，面向业务看板和数据 Agent 直接查询。

ADS 表通常是更宽、更好问、更适合业务口径的结果表：

- `ads_buy_channel_roi_daily`：买量 ROI 日报。
- `ads_material_daily`：素材效果日报。
- `ads_cohort_pivot`：同期群宽表，把 D1/D3/D7/D30/D360 等节点摊平成列。
- `ads_op_game_overview_daily`：运营游戏日报，不拆广告维度，更适合运营总览。

ADS 更适合回答业务问题，例如：

- 最近 30 天哪个广告平台 ROI 最高？
- 哪个素材 30 日 LTV 最好？
- 每个游戏的 DAU、收入、ARPU、D7 留存是多少？
- 哪个渠道的买量新增最多？


## 表 1：ads_buy_channel_roi_daily

### 表定位

`ads_buy_channel_roi_daily` 是买量 ROI 日报表。

它面向买量团队，用于分析广告渠道、投放团队、广告平台、广告层级带来的新增、收入、留存和 LTV。

### 数据粒度

一行表示：

某一天 + 某游戏 + 某渠道 + 某投放团队 + 某广告平台 + 某广告层级组合 的 ROI 指标。

核心维度：

- `stat_date`：统计日期，也是归因基准日。
- `papp_id`：主游戏 ID。
- `app_id`：子游戏 ID。
- `cch_id`：渠道商 ID。
- `bus_company_id`：业务公司。
- `ptid`：父团队 ID。
- `rctid`：子团队 ID。
- `md_vid`：广告投放者账号。
- `ad_pmedia`：广告平台，例如 bytedance、tencent、meta、google。
- `tg_level_1`、`tg_level_2`、`tg_level_3`：广告层级，通常对应计划、广告组、创意。

### 核心指标

- `spend`：广告消耗，单位是元。
- `platform_new_user_cnt`：平台新用户数。
- `game_new_user_cnt`：游戏新用户数。
- `new_device_cnt`：新增设备数。
- `new_role_cnt`：新建角色数。
- `pay_user_cnt`：付费用户数。
- `first_pay_user_cnt`：首次付费用户数。
- `pay_amt_sum`：当日收入，单位是分。
- `retain_d1_cnt`、`retain_d7_cnt`、`retain_d30_cnt`、`retain_d90_cnt`：不同周期留存人数。
- `ltv_1d`、`ltv_7d`、`ltv_30d`、`ltv_360d`：不同周期 LTV，单位是分。
- `max_ltv_day_filled`：当前已经回填到哪个 LTV 周期。
- `is_spend_synced`：广告消耗是否已经同步。

### 常见计算

ROI：

```sql
SUM(pay_amt_sum) / 100 / NULLIF(SUM(spend), 0)
```

D7 ROI：

```sql
SUM(ltv_7d) / 100 / NULLIF(SUM(spend), 0)
```

D30 ROI：

```sql
SUM(ltv_30d) / 100 / NULLIF(SUM(spend), 0)
```

D7 留存率：

```sql
SUM(retain_d7_cnt) / NULLIF(SUM(game_new_user_cnt), 0)
```

首付转化率：

```sql
SUM(first_pay_user_cnt) / NULLIF(SUM(game_new_user_cnt), 0)
```

### 适合数据 Agent 回答的问题

- 最近 30 天哪个广告平台 ROI 最高？
- bytedance 在 2026 年 5 月的消耗、收入、ROI 分别是多少？
- 哪个投手账号 `md_vid` 带来的 D7 ROI 最高？
- 哪个广告层级的新增用户最多？
- D30 ROI 低于 1 的投放组合有哪些？

### 与其他表的关系

- 可以和 `dws_v4_ad_cost_daily` 对比广告消耗口径。
- 可以和 `ads_material_daily` 对比素材维度效果。
- 可以和 `ads_cohort_pivot` 对比更完整的 D1-D360 留存与 LTV。


## 表 2：ads_cohort_pivot

### 表定位

`ads_cohort_pivot` 是同期群透视宽表。

它把一个新增队列从 D0 到 D360 的留存、付费留存、LTV、累计付费用户等指标摊平成一行，方便数据 Agent 直接查询。

### 数据粒度

一行表示：

某个进入日期 + 某游戏 + 某渠道 + 某广告维度 + 某用户类型 的完整生命周期队列。

核心维度：

- `enter_date`：新增日期，也是队列起点。
- `papp_id`、`app_id`、`cch_id`：游戏和渠道维度。
- `bus_company_id`：业务公司。
- `is_platform_new_user`：是否平台新用户。
- `ad_pmedia`：广告平台。
- `ad_cch_id`：广告渠道 ID。
- `tg_level_1`、`tg_level_2`、`tg_level_3`：广告层级。
- `ad_material_id`：广告素材 ID。
- `ad_material_type`：素材类型。
- `ad_position`：广告版位。
- `ad_link_id`：广告链路 ID。

### 核心指标

队列规模：

- `enter_user_cnt`：D0 新增用户数。
- `enter_device_cnt`：D0 新增设备数。
- `enter_pay_user_cnt`：D0 付费用户数。
- `enter_pay_device_cnt`：D0 付费设备数。

活跃留存：

- `retain_d1_cnt`
- `retain_d3_cnt`
- `retain_d7_cnt`
- `retain_d14_cnt`
- `retain_d30_cnt`
- `retain_d60_cnt`
- `retain_d90_cnt`
- `retain_d180_cnt`
- `retain_d360_cnt`

设备留存：

- `d_retain_d1_cnt`
- `d_retain_d7_cnt`
- `d_retain_d30_cnt`
- `d_retain_d90_cnt`
- `d_retain_d360_cnt`

付费留存：

- `pay_retain_d1_cnt`
- `pay_retain_d7_cnt`
- `pay_retain_d30_cnt`
- `pay_retain_d360_cnt`

LTV：

- `ltv_1d`
- `ltv_3d`
- `ltv_7d`
- `ltv_14d`
- `ltv_30d`
- `ltv_60d`
- `ltv_90d`
- `ltv_180d`
- `ltv_360d`

累计付费：

- `cumulative_pay_user_d7`
- `cumulative_pay_user_d30`
- `cumulative_pay_user_d90`
- `cumulative_pay_user_d360`
- `cumulative_pay_device_d30`
- `cumulative_pay_device_d90`
- `cumulative_pay_device_d360`

回填状态：

- `max_day_filled`：当前回填到第几天。
- `is_finalized`：是否已经完整回填。
- `last_update_ts`：最后更新时间。

### 常见计算

D7 留存率：

```sql
SUM(retain_d7_cnt) / NULLIF(SUM(enter_user_cnt), 0)
```

D30 留存率：

```sql
SUM(retain_d30_cnt) / NULLIF(SUM(enter_user_cnt), 0)
```

D30 人均 LTV：

```sql
SUM(ltv_30d) / 100 / NULLIF(SUM(enter_user_cnt), 0)
```

D360 人均 LTV：

```sql
SUM(ltv_360d) / 100 / NULLIF(SUM(enter_user_cnt), 0)
```

D30 累计付费率：

```sql
SUM(cumulative_pay_user_d30) / NULLIF(SUM(enter_user_cnt), 0)
```

### 适合数据 Agent 回答的问题

- 哪个广告平台 D30 留存率最高？
- 最近三个月各游戏的 D7、D30、D90 留存趋势如何？
- 哪个素材 D360 LTV 最好？
- 平台新用户和老用户试新游的 LTV 差异是多少？
- 付费留存和活跃留存哪个渠道更稳定？

### 与其他表的关系

- 比 `dws_v4_cohort_user` 更适合直接问业务问题。
- 比 `ads_buy_channel_roi_daily` 更完整，因为它包含更多生命周期节点。
- 可与 `ads_material_daily` 在素材维度上互相校验。

状态：

```sql
SUM(pay_amt_sum) / 100 / NULLIF(SUM(dau), 0)
```

- 如果问题涉及小时级明细，应转向 `dws_v4_user_daily` 或 `dws_v4_device_daily`。


## 表 5：dws_v4_ad_cost_daily

### 表定位

`dws_v4_ad_cost_daily` 是 DWS V4 广告消耗表。

它保存广告平台同步回来的花费、曝光、点击、转化等指标，是 ROI 的成本侧来源。

### 数据粒度

一行表示：

某一天 + 某游戏 + 某广告追踪码 + 某广告平台 + 某广告账号 + 某广告层级 的花费数据。

核心维度：

- `stat_date`：广告花费发生日。
- `papp_id`、`app_id`、`cch_id`：游戏和渠道。
- `md_traceid`：广告追踪码。
- `ad_pmedia`：广告平台。
- `ad_cch_id`：广告渠道 ID。
- `ad_account_id`：广告账号 ID。
- `tg_level_1`、`tg_level_2`、`tg_level_3`：广告层级。
- `ad_material_id`：广告素材 ID。
- `ad_position`：广告版位。
- `currency`：货币。

### 核心指标

- `cost_amt`：广告花费，单位是分。
- `impression_cnt`：曝光次数。
- `click_cnt`：点击次数。
- `platform_convert_cnt`：广告平台上报转化数。
- `is_finalized`：数据状态，0 表示临时值，1 表示 T+1 精确，2 表示最终结算。
- `sync_ts`：广告 API 同步时间戳。

### 常见计算

CTR：

```sql
SUM(click_cnt) / NULLIF(SUM(impression_cnt), 0)
```

CVR：

```sql
SUM(platform_convert_cnt) / NULLIF(SUM(click_cnt), 0)
```

CPM：

```sql
SUM(cost_amt) / 100 / NULLIF(SUM(impression_cnt), 0) * 1000
```

CPC：

```sql
SUM(cost_amt) / 100 / NULLIF(SUM(click_cnt), 0)
```

CPA：

```sql
SUM(cost_amt) / 100 / NULLIF(SUM(platform_convert_cnt), 0)
```

### 适合数据 Agent 回答的问题

- 最近 30 天哪个广告平台消耗最多？
- 哪个广告账号 CTR 最高？
- 哪个广告层级 CPA 最低？
- 哪些广告还没有最终结算？
- 某个素材的点击率和转化率是多少？

### 与其他表的关系

- 可以与 `ads_buy_channel_roi_daily` 共同计算 ROI。
- 可以与 `dws_v4_user_daily` 按 `md_traceid`、广告层级、日期关联，分析消耗到注册/付费的转化。
- 可以与 `ads_material_daily` 按素材维度关联，分析素材花费与 LTV。


## 表 6：dws_v4_cohort_device

### 表定位

`dws_v4_cohort_device` 是设备队列表。

它从设备视角追踪一个新增设备队列在 D0 到 D360 的留存、付费和累计 LTV。

### 数据粒度

一行表示：

某个设备进入日期 + 某小时 + 某游戏 + 某广告归因维度 + 某个 `day_n` 的设备队列指标。

核心维度：

- `enter_date`：设备进入日期。
- `enter_hour`：设备进入小时。
- `papp_id`、`app_id`、`cch_id`：游戏和渠道。
- `app_type`：游戏类型。
- `md_traceid`：广告追踪码。
- `ad_cch_id`：广告渠道 ID。
- `ad_account_id`：广告账号 ID。
- `tg_level_1`、`tg_level_2`、`tg_level_3`：广告层级。
- `ad_material_id`：广告素材 ID。
- `ad_position`：广告版位。
- `day_n`：进入后第 N 天。

### 核心指标

队列基数：

- `d0_enter_device_cnt`：D0 进入设备数。
- `d0_pay_device_cnt`：D0 付费设备数。
- `d0_pay_amt`：D0 充值金额，单位分。


```sql

```

## 表 9：dws_v4_user_daily

### 表定位

`dws_v4_user_daily` 是用户日流量表。

它按小时统计登录、新增进入、注册、创角、付费、首付等指标，是 DWS V4 中最重要的用户事实汇总表之一。

### 数据粒度

一行表示：

某一天 + 某小时 + 某游戏 + 某广告归因维度 的用户流量和付费指标。

核心维度：

- `stat_date`：统计日期。
- `stat_hour`：统计小时。
- `papp_id`、`app_id`、`cch_id`：游戏和渠道。

- `ad_position`：广告版位。

- `login_user_cnt`：登录用户数，也可理解为小时级 DAU 分片。
- `login_device_cnt`：登录设备数。
- `login_user_bitmap`：登录用户 Bitmap，占位字段。

- `new_enter_user_cnt`：新增进入用户数。
- `new_enter_device_cnt`：新增进入设备数。
- `reg_user_cnt`：注册用户数。
- `reg_device_cnt`：注册设备数。

### 常见计算


```sql

```

创角率：

```sql
SUM(new_enter_role_cnt) / NULLIF(SUM(new_enter_user_cnt), 0)
```

## 字段单位与口径提醒

### 金额单位

需要特别注意：

- `ads_buy_channel_roi_daily.spend` 的单位是元。
- 大多数收入、LTV、成本字段的单位是分。
- 例如 `pay_amt_sum`、`ltv_30d`、`cost_amt`、`trans_amt` 通常需要除以 100 转成元。

常见转换：

```sql
SUM(pay_amt_sum) / 100
SUM(trans_amt) / 100
SUM(ltv_30d) / 100
SUM(cost_amt) / 100
```

### 留存率口径

用户留存率：

```sql
SUM(retain_user_cnt) / NULLIF(SUM(d0_enter_user_cnt), 0)
```

ADS 宽表留存率：

```sql
SUM(retain_d7_cnt) / NULLIF(SUM(enter_user_cnt), 0)
```

运营日报留存率：

```sql
SUM(retain_d7_cnt) / NULLIF(SUM(game_new_user_cnt), 0)
```

### ROI 口径

当日 ROI：

```sql
SUM(pay_amt_sum) / 100 / NULLIF(SUM(spend), 0)
```

长期 ROI：

```sql
SUM(ltv_30d) / 100 / NULLIF(SUM(spend), 0)
```

如果使用 `dws_v4_ad_cost_daily.cost_amt`：

```sql
SUM(revenue_amount_in_cent) / NULLIF(SUM(cost_amt), 0)
```

因为二者都是分，可以不除以 100。

### Bitmap 字段提醒

以下字段在 MySQL 中是 `LONGBLOB`：

- `*_bitmap`
- `user_bitmap`

它们是为了兼容 Doris 表结构而保留。

当前 MySQL 演示数据中，这些字段是模拟二进制值，不适合直接做 Bitmap 计算。

数据 Agent 生成 SQL 时，应优先使用对应的计数字段，例如：

- `login_user_cnt`
- `new_device_cnt`
- `retain_user_cnt`
- `cum_pay_user_cnt`
- `user_cnt`
